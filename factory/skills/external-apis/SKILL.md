# External APIs — Dual-Path Contract

Same rule as `stripe-checkout` and `vercel-neon`: **every integration ships in two modes.** Mock by default, real when its env var is set. The site never crashes for lack of keys; the repo owner upgrades in minutes.

Apply this pattern to:

- **LLM calls** — OpenAI, Anthropic, Vercel AI Gateway (recommended entry point).
- **Email** — Resend, SendGrid.
- **Images / transcription / embeddings** — same shape, different provider.
- **Any third-party HTTP API that requires a key.**

Do NOT apply this to services with no keys (public APIs, open data) — call them directly.

## The branch — LLM example

Vercel AI Gateway is the recommended path: **one key (`AI_GATEWAY_API_KEY`), any model**. Pass a model string like `'openai/gpt-4o-mini'` or `'anthropic/claude-4.5-sonnet'` directly to `generateText` / `streamText` from the `ai` package — the SDK auto-detects `AI_GATEWAY_API_KEY` and routes through the gateway. No separate `@ai-sdk/openai` or `@ai-sdk/anthropic` imports needed for the gateway path.

```ts
// lib/llm.ts — one file, two paths. Copy this shape for any external API.
//
// When AI_GATEWAY_API_KEY is set (locally via `vercel env pull`, in
// production via the devops subagent pushing it), `ai`'s streamText/
// generateText route through Vercel AI Gateway using the `provider/model`
// string. When it isn't set, we serve canned replies in-character.

export type LLMSource = 'gateway' | 'mock';
export type LLMReply = { text: string; source: LLMSource };

const MODEL = 'openai/gpt-4o-mini'; // swap per product voice

export async function complete(prompt: string): Promise<LLMReply> {
  if (!process.env.AI_GATEWAY_API_KEY) {
    return { text: cannedReply(prompt), source: 'mock' };
  }
  try {
    const { generateText } = await import('ai');
    const { text } = await generateText({ model: MODEL, prompt });
    return { text, source: 'gateway' };
  } catch (err) {
    console.warn('[llm] gateway path failed, falling back to mock:', err);
    return { text: cannedReply(prompt), source: 'mock' };
  }
}

/** Streaming variant for UX where tokens appear as they arrive. */
export async function streamCompletion(prompt: string) {
  if (!process.env.AI_GATEWAY_API_KEY) {
    // Yield the canned reply one word at a time so the UI still feels
    // responsive in demo mode.
    return mockStream(cannedReply(prompt));
  }
  const { streamText } = await import('ai');
  const result = streamText({ model: MODEL, prompt });
  return result.textStream; // AsyncIterable<string>
}

function cannedReply(prompt: string): string {
  // Hand-written, stays in character. Rotate a few so the demo doesn't
  // look broken. Reference the business plan's voice.
  const replies = [
    'The eleven-minute cut of Oppenheimer is, frankly, an improvement.',
    'Our model has considered your request and filed it under "ambitious".',
    'Response generated by vibes. Wire AI_GATEWAY_API_KEY for real text.',
  ];
  return replies[prompt.length % replies.length];
}

async function* mockStream(text: string): AsyncGenerator<string> {
  for (const word of text.split(' ')) {
    await new Promise((r) => setTimeout(r, 60));
    yield word + ' ';
  }
}
```

### Route handler — stream tokens to the browser

```ts
// app/api/chat/route.ts
import { streamCompletion } from '@/lib/llm';

export const runtime = 'nodejs';

export async function POST(req: Request): Promise<Response> {
  const { prompt } = (await req.json()) as { prompt: string };
  const stream = await streamCompletion(prompt);

  const encoder = new TextEncoder();
  const body = new ReadableStream<Uint8Array>({
    async start(controller) {
      for await (const chunk of stream) controller.enqueue(encoder.encode(chunk));
      controller.close();
    },
  });
  return new Response(body, {
    headers: { 'content-type': 'text/plain; charset=utf-8' },
  });
}
```

### One-shot variant — await the whole response

```ts
import { complete } from '@/lib/llm';

const { text, source } = await complete('Invent a holiday.');
// source === 'gateway' when AI_GATEWAY_API_KEY is set, else 'mock'.
```

### Quick reference: model strings

The gateway accepts `<provider>/<model>` — the `ai` package passes it straight through. Pick whichever fits the startup's voice and budget:

| String | Notes |
|---|---|
| `openai/gpt-4o-mini` | Cheap, fast. Default for most generated sites. |
| `openai/gpt-4o` | Heavier, better reasoning. |
| `anthropic/claude-4.5-sonnet` | Strong writing, higher cost. |
| `anthropic/claude-haiku-4-5-20251001` | Cheap + fast, Anthropic flavor. |

Full list: [vercel.com/ai-gateway/models](https://vercel.com/ai-gateway/models).

### Key rules

- **Dynamic imports.** `await import('ai')` inside the branch so `npm run build` succeeds with zero SDKs installed and zero env vars.
- **Wrap each `await import` in try/catch.** A missing package falls through to the mock instead of 500ing.
- **Expose a `source` field.** Let the UI render "Demo response" in mock mode — honesty plays better than fake confidence.
- **Never throw** on missing credentials. A missing key is the default state, not an error.
- **Gateway-first.** Start every branch chain with `AI_GATEWAY_API_KEY`. Only add direct `OPENAI_API_KEY` / `ANTHROPIC_API_KEY` branches if the product has a reason to bypass the gateway (privacy, latency, provider-specific features).

## `.env.example`

Every external API the site touches gets a section in `.env.example`. Always commented out. Always include the exact URL to get the key, plus one-line guidance on cost/free-tier.

```env
# ──────────────────────────────────────────────────────────────
# LLM (optional — canned responses by default)
# Option A — Vercel AI Gateway. Recommended: unified billing across
# all providers, auto-provisioned by `vercel integration add ai-gateway`.
# Free starter credits, then metered. Keys auto-inject into Vercel env.
# https://vercel.com/marketplace/ai-gateway
# AI_GATEWAY_API_KEY=...
#
# Option B — direct provider keys (pick one).
# OpenAI: https://platform.openai.com/api-keys
# OPENAI_API_KEY=sk-...
# Anthropic: https://console.anthropic.com/settings/keys
# ANTHROPIC_API_KEY=sk-ant-...
# ──────────────────────────────────────────────────────────────

# ──────────────────────────────────────────────────────────────
# Email (optional — console.log by default)
# Resend: https://resend.com/api-keys (free tier: 100/day, 3000/mo)
# RESEND_API_KEY=re_...
# ──────────────────────────────────────────────────────────────
```

Sections go in the same `.env.example` as `stripe-checkout` and `vercel-neon` — one file, every optional upgrade documented, all commented out. The file is the upgrade manual.

## `package.json`

Gateway-only path needs ONE dependency — the `ai` package handles everything via model strings:

```json
{
  "dependencies": {
    "ai": "^5.0.0"
  }
}
```

Only add direct provider packages (`@ai-sdk/openai`, `@ai-sdk/anthropic`) if the site has a second branch that bypasses the gateway. Most generated sites don't — the gateway covers every provider with one key. If `ai` itself fails to install, the try/catch around `await import('ai')` falls through to the canned branch — `npm run build` still succeeds.

## When Vercel gives it to you free

Some things the Vercel Marketplace auto-provisions into every project env so you literally don't need to do anything:

| Service | Env var | How it's wired |
|---|---|---|
| Neon Postgres | `DATABASE_URL` | `vercel integration add neon` (see `vercel-neon` skill) |
| Vercel AI Gateway | `AI_GATEWAY_API_KEY` | `vercel integration add ai-gateway` |
| Vercel Blob | `BLOB_READ_WRITE_TOKEN` | automatic once the project has a Blob store |
| Vercel KV / Upstash Redis | `KV_*` | `vercel integration add upstash` |

For these four: the dual-path code still runs locally with a mock (because the env var is missing in `.env.local` until you `vercel env pull`), but a production deploy to Vercel picks up the real key with zero per-project setup.

Everything NOT in that table (Stripe, OpenAI, Anthropic direct, Resend, Twilio, …) needs a human to paste a key into the Vercel dashboard or `.env.local`. Document that path in the README section but still SHIP with the mock active.

## Testing the mock works

```bash
npm run dev
# Call whatever route uses lib/llm.ts, e.g.:
curl -X POST http://localhost:3000/api/chat \
  -H 'content-type: application/json' \
  -d '{"prompt":"why is this a company?"}'
# Expect a response with `"source":"mock"` and a hand-written reply.
```

Before marking any external-API step complete:
- Verify `npm run build` succeeds with **no** `*_API_KEY` env vars set.
- Verify the UI does not show an error state in mock mode — it should look
  like the feature works, because it does (just with canned output).
- Verify `tsc --noEmit` is clean.
- If the feature is visibly "AI-powered" in the product copy, add a small
  "Demo response" badge in mock mode so nobody thinks the live model is
  the one producing hand-written replies.

## When NOT to use this skill

- The feature has no external dependency (pure UI, static data).
- The external dependency has no credentials (public API, open endpoint).
- The feature is so central that a canned fallback would be dishonest
  (e.g. "AI summarizer" where the summary is the product). In that case,
  either make the key required with a clear error, OR don't ship the
  feature in the MVP.
